# ChatCC vs OpenClaw 详细对比

[OpenClaw](https://github.com/openclaw/openclaw) 是一个运行在本地设备上的通用个人 AI 助手，支持 20+ 消息平台（WhatsApp、Telegram、Slack、Discord、Feishu 等）。ChatCC 则专注于通过飞书远程控制本地 Claude Code。

两者定位不同，可以互补使用。

## 核心区别

| 维度 | OpenClaw | ChatCC |
|------|----------|--------|
| **定位** | 通用个人 AI 助手，支持多平台 | 专注飞书的 Claude Code 远程控制网关 |
| **核心功能** | 多模型 AI 对话（ChatGPT/Claude/Gemini） | Claude Code 专用执行环境（本地开发工具） |
| **消息平台** | 20+ 平台（WhatsApp/Telegram/Slack 等） | 仅飞书（深度集成） |
| **使用场景** | 日常 AI 对话、跨平台统一助手 | 远程控制本地开发环境、执行 Claude Code 任务 |
| **技术栈** | Node.js | Go（单二进制部署） |
| **会话管理** | AI 对话历史管理 | tmux 持久会话 + 交互式命令执行 |
| **本地执行** | 主要调用 AI API | 直接在本地执行 Claude Code（访问文件系统） |
| **扩展性** | 技能系统（Skills） | 命令系统（Shell、项目管理等） |
| **配置复杂度** | 多平台 + 多模型认证 | 仅需飞书 app_id / app_secret |
| **目标用户** | 需要跨平台 AI 助手的个人用户 | 需要远程控制开发环境的开发者 |

## 何时选择 ChatCC？

1. **专注 Claude Code 远程执行** — 通过飞书远程控制本地 Claude Code，访问本地文件系统，执行代码分析和自动化任务，支持长时间运行的会话（tmux 持久化）。

2. **开发者工作流集成** — 在飞书中查询项目代码、执行脚本、处理交互式提示（y/n 确认等）。

3. **企业内网环境** — 已用飞书协作，需要从外部访问内网开发机，不想暴露额外端口。

4. **轻量部署** — 单平台、单二进制、5 分钟部署，仅需飞书应用配置。

5. **嵌套会话** — 在 Claude Code 环境中启动子进程，已解决环境变量过滤问题。

## 何时选择 OpenClaw？

1. **跨平台统一助手** — 在 WhatsApp、Telegram、Slack 等多平台使用同一个 AI 助手。

2. **多模型 AI 对话** — 同时使用 OpenAI、Claude、Gemini 等模型，支持故障转移和自动切换。

3. **通用 AI 能力** — 日常对话、技能系统扩展，不限于开发工具。

4. **语音和多模态** — 语音输入/输出、Canvas 界面等丰富交互方式。

5. **Node.js 生态** — 团队熟悉 Node.js/TypeScript，希望二次开发。

## 技术优势对比

### ChatCC

| 优势 | 说明 |
|------|------|
| Claude Code 专用优化 | 环境变量过滤、嵌套会话支持、交互式提示检测 |
| 开发工具集成 | Shell 白名单、项目别名、文件系统直接访问 |
| 极简配置 | 仅需飞书应用配置，无需多平台认证 |
| 本地优先 | 直接访问本地文件系统，无网络延迟，支持内网资源 |
| Go 性能 | 单二进制、低内存、快速启动 |

### OpenClaw

| 优势 | 说明 |
|------|------|
| 跨平台统一 | 20+ 消息平台，统一界面，无缝切换 |
| 多模型支持 | 多 AI 提供商、自动故障转移、任务级模型选择 |
| 丰富交互 | 语音 I/O、Canvas 渲染、多模态 |
| 社区生态 | 活跃的开源社区、丰富的技能系统 |

## 互补使用

两个项目服务于不同需求，完全可以同时使用：

- **OpenClaw** — 日常 AI 助手，处理通用对话、跨平台消息
- **ChatCC** — 开发工具网关，处理 Claude Code 相关的开发任务

```
WhatsApp/Telegram ──→ OpenClaw ──→ 日常 AI 对话
飞书             ──→ ChatCC   ──→ 远程控制 Claude Code
```

各司其职，互不干扰。
